## Workflow

### CREATE Mode (`.design` file does not exist)

1. **Create directory structure**: Create `assets/`, `pages/`, and `partials/` under `{project-name}/`. `partials/` stores shared HTML fragments generated by the Step 3 generation tree before leaf pages are assembled. `theme/` is legacy-only and is not created in the CSS-only primary path.
2. **Prepare brand CSS**: Use the selected Design Library `colors_and_type.css`, or create a project-local `colors_and_type.css` in the project root when no Library is active.
3. **Create `.design` canvas entry file**: Main Agent writes all page skeleton nodes in one pass before dispatching page Sub-Agents. Set `config.autoLayout` to `true`, `config.deviceType` to the selected device, and `config.projectName` to the user-friendly project name in the user's language (e.g., Chinese user → `"咖啡品牌官网"`).
4. **Generate shared fragments and pages by generation tree**: Main Agent writes `generation-tree.json` in the project root, mirrors it into `orchestration-summary.json.project.generationTree`, generates shared `partials/*.html` first, then dispatches leaf Sub-Agents to generate only `pages/*.html` from inherited fragments. Sub-Agents must not read, write, append, or validate `.design`.
5. **Register every image asset** ([REQUIRED]): After all image files exist under `assets/`, the Main Agent appends one `type: "image"` node into `.design` `data` array **per asset file** (matching extensions `.jpg/.jpeg/.png/.gif/.webp/.svg`). See `workflows/create-project.md` Step 2.5d for the full procedure. **No image file in `assets/` may go unregistered.**
6. **Register wiring and validate**: After all Sub-Agents complete, Main Agent updates page `interactions` and executes `scan-design-directory.mjs`.

### UPDATE Mode (`.design` file already exists)

**Editing existing pages**: Only when the user explicitly requests in-place overwrite/modification, directly modify `pages/*.html` and do not touch the `.design` file. Otherwise, all page-level or element-level changes to an existing/current/selected page are comparison-first: append/insert a new page node with a distinct `htmlSrc` and keep the source page immutable. Selected element/xpath/`design_page` context is only a source locator; it never implies permission to overwrite the source page. Button behavior, click interactions, state changes, visual tweaks, and "this part" edits must still create a new comparison page unless the user explicitly says not to add one.

**Adding new pages**:
1. Main Agent pre-assigns page IDs, `htmlSrc`, and `group`, then appends new page skeleton node(s) to `.design`.
2. Main Agent updates `generation-tree.json` when multiple new pages/states share structure. Shared fragments are generated under `partials/`; leaf Sub-Agents generate HTML files in `pages/` only.
3. Main Agent optionally registers wiring requested by the user and executes `scan-design-directory.mjs` to verify the full project.

**Adding new image assets** ([REQUIRED]): When any new file is added under `assets/` (whether part of a page addition flow or a standalone "Add assets" request), the Main Agent must append one `type: "image"` node into `.design` `data` for each new image file. See `workflows/edit-project.md` Step 2b' for the full procedure. Image assets that exist on disk but are not registered as image nodes will never surface on the canvas.

**Deleting pages**: Main Agent removes the target page node, deletes the corresponding HTML file, and removes any incoming/outgoing `interactions` that reference the deleted page ID.

**Deleting image assets**: Remove the corresponding `type: "image"` node from the `data` array and delete the image file from `assets/`. Keep the project image-id counter monotonic — do not reuse the deleted node's id for new assets.

## Directory Structure

```
<project-name>/
├── <project-name>.design   # Canvas entry (DesignFileData JSON)
├── generation-tree.json     # Materialized shared/private generation dependency tree
├── assets/                  # Static resources (images, etc.), flat structure
├── partials/                # Shared HTML fragments generated before leaf pages
├── pages/                   # HTML page files
└── colors_and_type.css       # Project-local brand CSS when no external Design Library is used
```

## Naming Rules

| Type | Rule | Example |
|------|------|---------|
| Project name | kebab-case, matches `.design` filename | `my-portfolio/my-portfolio.design` |
| Page file | kebab-case, semantic, no numeric prefixes | `about-us.html`, `product-detail.html` |
| Asset file | Semantic kebab-case, no numeric prefixes or suffixes; icons use `icon-` prefix | `hero-background.jpg`, `team-collaboration.png`, `icon-menu.svg` |
| Legacy theme file | kebab-case + `.theme` suffix; only for old projects containing theme nodes | `modern-light.theme` |

### Legacy Theme Naming Rules

Theme nodes and `.theme` files are legacy compatibility artifacts. New CSS-only projects do not create them. When repairing or migrating an old project that already contains theme nodes, preserve the existing `sourceId` mapping and use the project name as a business prefix for any newly created legacy `.theme` filename:

```
{project-name}-{library-theme-name}.theme
```

Example: project name `coffee-brand`, Library theme name `aurora-indigo` → filename is `coffee-brand-aurora-indigo.theme`.

The corresponding `ThemeData.name` field must match the filename (without `.theme` suffix), i.e., `coffee-brand-aurora-indigo`.

**Purpose**: The business prefix prevents conflicts in legacy projects. Do not introduce `.theme` files into new CSS-only projects.

## Type Definitions

```typescript
type FileNodeType = 'page' | 'theme' | 'image'; // theme is legacy optional
type DeviceType = 'desktop' | 'mobile' | 'tablet' | 'freeSize';

interface DesignFileData {
  data: Node[];
  config?: DesignFileConfig;
}

interface DesignFileConfig {
  autoLayout: boolean;
  deviceType: DeviceType;
  projectName?: string;  // User-friendly project display name in user's language (e.g., "咖啡品牌官网")
  designLibrary?: DesignLibraryIdentity | null;  // Required in Library-bound mode; null/omitted in free-explore mode
}

interface DesignLibraryIdentity {
  name: string | null;          // Design Library display name from context or metadata
  id: string | null;            // Stable Design Library id from context/API when available
  version: string | number | null; // Selected/current Library version; null only when unavailable
  scope: string | null;         // workspace | custom-global | built-in-global | other source scope
  path: string | null;          // Absolute Library directory path when available
  versionSource: string | null; // selected-context | metadata.json | library-consumption.json | package.json | api | null
}

interface Node {
  id: string;
  title: string;
  type: FileNodeType;
  version: number;       // Node version number, defaults to 1 on creation
  createdAt: number;
  devMetadata: DevPageData | DevThemeData | DevImageData;
  canvasData: FileCanvasData | FileImageCanvasData;
}

interface DevPageData {
  htmlSrc: string;            // Relative path, e.g. "pages/index.html"
  interactions?: FileElementInteraction[];
  supersedesPageId?: string;  // Optional. This page is the replacement version of the referenced page node id. Writing it declares the old page "retired": the old page stays on the canvas for comparison but must no longer participate in preview navigation (see wiring-strategy.md "Version Convergence")
}

interface FileElementInteraction {
  domId: string;             // Value of the data-dom-id attribute on the HTML element
  targetPageId: string;      // Target page node ID to navigate to on click
  hideEdge?: boolean;        // When true, preview still navigates but the canvas does not draw an edge. Use for auxiliary navigation such as Back, return-to-list, global nav, breadcrumbs, secondary CTAs, and skip links
  transitionLabel?: string;  // Transition logic description (e.g., "login success"). All interactions from the same source to the same target share the same label
}

// interactions example: "About Us" link in homepage navbar → navigates to page-002
// { "domId": "nav-about", "targetPageId": "page-002", "transitionLabel": "Learn more" }
// hidden return example: detail page Back button → navigates to list page without drawing a reverse canvas edge
// { "domId": "back-list", "targetPageId": "page-list", "hideEdge": true, "transitionLabel": "返回" }
// hidden shortcut example: homepage Blog CTA → navigates to blog page without drawing a Page 1 → Page 3 shortcut edge
// { "domId": "shortcut-blog", "targetPageId": "page-blog", "hideEdge": true, "transitionLabel": "Read blog" }

interface DevThemeData {
  sourceId: string;      // Legacy ThemeData.id, corresponds to theme/<sourceId>.theme
}

/** Image node dev metadata, references image files under assets/ directory */
interface DevImageData {
  imageSrc: string;      // Relative path to image file, e.g. "assets/hero-banner.png"
}

interface FileCanvasData {
  x: number;
  y: number;
  group: number;  // Group number, incrementing from 0. In CSS-only projects, groups separate style/version rows.
}

/** Image node canvas positioning data, does not include group field. Image nodes do not participate in grouping; in autoLayout they are placed after all groups. */
interface FileImageCanvasData {
  x: number;
  y: number;
}
```

### group Grouping Mechanism

The `group` field associates pages that belong to the same style/version row. The canvas SDK performs row-based layout by group:

| Rule | Description |
|------|-------------|
| Each group contains **N pages** in the CSS-only primary path | Legacy projects may also contain one theme node per group |
| Group numbers start from **0** incrementally | First style/version row is group=0, second row is group=1, etc. |
| Nodes in the same group are on the **same row** on canvas | Different groups have spacing between them, forming multi-row layout |
| Theme customization updates CSS in place | Do not create new groups for simple token changes; only variants/multi-style comparisons use additional groups |
| `interactions` `targetPageId` should point to pages **within the same group** | Cross-group navigation wiring is usually not needed |
| `interactions` `hideEdge` may only be boolean when present | `true` means the interaction is clickable in preview but excluded from visible canvas wiring |

### Design Library Identity

When `orchestration-summary.json.designSource.operatingMode === "library-bound"`, `.design.config.designLibrary` is required and must mirror `orchestration-summary.json.designSource.libraryIdentity`.

Rules:

- Main Agent writes `config.designLibrary` during `.design` initialization before page Sub-Agents are dispatched.
- Preserve `config.designLibrary` on edits, page additions, redesigns, and repairs unless the user explicitly switches Design Library.
- If a field is unavailable, write `null`; do not omit the field.
- `version` may be a string or number. If no selected/context version is provided, read Library metadata files and common version fields (`version`, `libraryVersion`, `packageVersion`, `releaseVersion`, `currentVersion`, `current_version`). If still unavailable, set `version: null`.
- In free-explore mode, `designLibrary` should be `null` or omitted.
- This field is traceability metadata only. Sub-Agents must still use the actual Library files and token references passed in `orchestration-summary.json`.

### version Field

`Node.version` is a positive integer and defaults to `1` on creation. Legacy `ThemeData.version` only applies when repairing old projects that still contain `.theme` files.

## Node Templates

### Page Node

```json
{
  "id": "page-<slug>",
  "title": "<page title>",
  "type": "page",
  "version": 1,
  "createdAt": <timestamp>,
  "devMetadata": {
    "htmlSrc": "pages/<slug>.html",
    "interactions": [
      {
        "domId": "nav-about",
        "targetPageId": "page-about"
      }
    ]
  },
  "canvasData": { "x": 0, "y": 0, "group": 0 }
}
```

- **ID Rule**: `page-{slug}` format, where `slug` is derived from the HTML filename minus the `.html` suffix (e.g., `pages/pricing.html` → `page-pricing`, `pages/about-us.html` → `page-about-us`). IDs must be unique within a project; when name conflicts occur, append a numeric suffix (e.g., if `page-index` exists, use `page-index-2`)
- **interactions**: Filled by the Main Agent after all Sub-Agents complete (see `operation-policies/wiring-strategy.md`). Sub-Agents keep it as empty array `[]` when creating page nodes
- **supersedesPageId** (optional): Set by the Main Agent when this page is a replacement version of an existing page (Quick Path A0/A1 comparison pages by default, unless the user explicitly asks to keep both versions usable). Must reference an existing page node id; no self-reference or cycles. Once set, Version Convergence is mandatory: redirect all incoming interactions from the superseded page to this page, and clear the superseded page's own `interactions` to `[]` (see `operation-policies/wiring-strategy.md` "Version Convergence")
- **canvasData**: Use 0 for x/y, autoLayout arranges automatically. `group` matches the style/version row.

### Theme Node (Legacy Optional)

```json
{
  "id": "theme-<slug>",
  "title": "<theme name>",
  "type": "theme",
  "version": 1,
  "createdAt": <timestamp>,
  "devMetadata": {
    "sourceId": "<theme-id>"
  },
  "canvasData": { "x": 0, "y": 0, "group": 0 }
}
```

- Theme nodes are not created in new CSS-only projects.
- In legacy projects that already contain theme nodes, `sourceId` must correspond to a `theme/<sourceId>.theme` file.
- SDK matches legacy theme files by sourceId (baseName matching preferred).

### Image Node

> **SSOT for the image node field template, id format, and title rule.** All workflows (`create-project.md` Step 2.5d, `edit-project.md` Step 2b', `redesign-ui.md` Step 4b, `customize-theme.md` Step 3.5.3) reference this spec when registering image nodes; they keep only their flow-specific steps (list new files → read `.design` → append → write back) and must not redefine node fields or title rules.

```json
{
  "id": "image-NNN",
  "title": "<image title>",
  "type": "image",
  "version": 1,
  "createdAt": <timestamp>,
  "devMetadata": {
    "imageSrc": "assets/<filename>.png"
  },
  "canvasData": { "x": 0, "y": 0 }
}
```

All fields above are required (`version` defaults to `1`; `createdAt` is the current timestamp in ms).

- **ID Rule**: `image-{NNN}` format, where NNN is a 3-digit zero-padded incrementing counter starting from `001` (e.g., `image-001`, `image-002`). Counter is **project-wide unique and monotonic** — it does not reset per group; when adding new assets to an existing project, scan existing image node IDs and continue from the largest one (do not reuse deleted IDs, do not restart from `001`)
- **imageSrc** must be a relative path pointing to a file under the `assets/` directory
- **title**: Must be a **semantic description meaningful in the user's most frequently used language** (see `user-facing-language.md` "Artifact Naming Language Rule"). E.g., for a Chinese user, `hero-main.jpg` → `"主视觉"`; for an English user, `hero-main.jpg` → `"Hero Main"`. **[FORBIDDEN] mechanically converting the ASCII filename to Title Case** (kebab-case → Title Case is NOT a valid title derivation).
- **canvasData** does not include the `group` field. Image nodes do not participate in group assignment; in autoLayout they are placed uniformly in the last row after all groups
- **[REQUIRED] Every image file under `assets/`** (extensions `.jpg/.jpeg/.png/.gif/.webp/.svg`) **must have a corresponding image node in `.design`** — banners, illustrations, brand images, decorative images, hero visuals, feature shots, etc. are all registered uniformly, not selectively. Non-image assets (fonts, json, etc.) are not registered. See `workflows/create-project.md` Step 2.5d and `workflows/edit-project.md` Step 2b' for the registration procedure

**Technical requirements**: All property names and string values use double quotes, no trailing commas, no comments.

## Validation

**Sub-Agents must not execute design project validation scripts.** They only perform local HTML self-checks and report completion. The **Main Agent** executes `scan-design-directory.mjs` before presenting results to the user; this script internally runs `.design` validation and directory consistency checks.

Validation items: JSON legality, `data` is a non-empty array, required node fields, `devMetadata` completeness, ID uniqueness, page/image file existence, legacy theme file existence when a theme node is present, page count match (when `--expected-pages` is passed), and interaction `domId` coverage.

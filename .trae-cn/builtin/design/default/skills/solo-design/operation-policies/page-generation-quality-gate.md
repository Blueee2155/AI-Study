# Page Generation Quality Gate

> **Audience**: Page Sub-Agents. Read before writing and execute after writing. This compact gate is mandatory for every page; deep aesthetic files are read only when triggered below.

---

## Gate 1 — Token Compliance

- [FORBIDDEN] Hardcoded colors in visible page content: `#hex`, `rgb(...)`, `hsl(...)`, `bg-[#...]`, `text-[#...]`, `border-[#...]`
- Data-identity colors (per-user avatars, data-viz hues, status indicators) in leaf-node inline `style` must NOT trigger any post-completion self-repair loop. If they remain at completion time, report completion once; the Main Agent validation/repair phase owns the final decision.
- [FORBIDDEN] Tailwind named color utilities: `bg-blue-*`, `text-gray-*`, `border-slate-*`, and equivalent palette classes
- [FORBIDDEN] Fabricating CSS variable names not listed in "Actual Token Name Reference"
- Use `var(--{tokenName})` or Library typography/radius/shadow tokens provided by Main Agent

Deep read trigger: If you are unsure about color role or contrast, read `aesthetics/index.md §4 Color & Surface`.

### Free-Explore Mode Token Compliance Relaxation

When `operatingMode === 'free-explore'` (no Design Library):
- Gate 1 token violations (hardcoded colors, named Tailwind colors) are **advisory, not blocking**.
- Sub-Agent should still prefer CSS variables, but must NOT enter a self-repair loop to fix aesthetic token issues.
- If a color value has no exact CSS variable match (e.g., star rating amber), use it as-is and report `"tokenRelaxation": ["<color-value>: <usage-reason>"]` in the completion report.
- [FORBIDDEN] Spending more than 1 tool call on token compliance fixes in free-explore mode.

## Gate 2 — Style Infrastructure

The final HTML must keep the head generated by `fill-html-head.mjs` intact:

- Verify every item in the `sub-agent-hard-rules.md §Style-Integrity` table (protected head elements + Lucide init script).
- [FORBIDDEN] Manually adding `<link rel="stylesheet">` for brand CSS.

Head management mode must be internally consistent (`SkeletonMainOnly` / `FullHtmlReplaceHead` defined in `sub-agent-hard-rules.md §Style-Integrity`; plus two Gate-2-specific modes): `DerivedCopyMutableOnly` — derived state copied a base HTML file and edited only mutable regions; `SharedFragmentOnly` — shared/parent fragment output contains no `<html>`, `<head>`, or `<body>`.

[FORBIDDEN] `fill-html-head.mjs` skeleton generation followed by full-file `Write` on the same HTML file. If completion evidence indicates this mixed mode, or if `htmlWriteMode` / `headManagementEvidence` is missing, Gate 2 fails.

---

## Gate 3 — Library Conformance

When a Design Library exists, verify compliance with `sub-agent-hard-rules.md §Component-Conformance` (full reading rules SSOT): every planned `componentPlan[]` preview/contract file was read and implemented from; the assigned UI Kit `index.html` was read when provided; at most 2 extra resolved component contracts were read and reported.

[FORBIDDEN] Generic implementation of a planned/discovered component when a resolved component contract exists.

---

## Gate 3b — Icon Source Awareness (Advisory Only — NEVER blocks, NEVER triggers repair)

> This gate is purely informational. It does NOT block page completion, does NOT trigger self-repair, and does NOT require re-generation. Its only purpose is transparency in the completion report.

When `designSource.iconAssets` exists in orchestration-summary:

- Completion report SHOULD include `libraryIconsUsed` array and `lucideFallbackIcons` array
- [ADVISORY] If the page uses icons and `libraryIconsUsed` is empty → note in report (not actionable by Sub-Agent)
- [FORBIDDEN] Self-repair loop to replace Lucide icons with Library icons after page is written
- [FORBIDDEN] Re-generating page content because of icon source choices
- [FORBIDDEN] Treating icon source as a blocking gate that prevents completion

> **Design rationale**: Icon source priority is a GENERATION-TIME directive (§Icon-Source-Priority guides what to write); it is NOT a post-generation enforcement mechanism. If the Sub-Agent happened to use Lucide for an icon that Library also has, this is a mild miss — the page works correctly either way. The Main Agent MAY perform a lightweight targeted fix (single Edit replacing one tag) during 3.5a-1 consolidation if it deems necessary, but this is Main Agent's discretion and never Sub-Agent's self-repair responsibility.

## Gate 4 — Responsive Layout

> **[Skip condition]**: When `orchestration-summary.json.project.dashboardMode === true`, this gate is skipped — dashboard pages use fixed viewport layout by design.

All multi-column layouts need mobile-first fallbacks; image-text layouts stack on small screens; fixed pixel widths that can cause horizontal overflow are forbidden; mobile cards must not squeeze auxiliary rounded blocks beside primary text without `md:` or larger breakpoint.

Deep read trigger: For mobile pages or dense cards, read `html-implementation-mobile.md`; for desktop/tablet complex columns, read `html-implementation-web.md`.

---

## Gate 5 — Image Integrity (Script-Backed — Report Only)

> **Script backstop**: Image path validity (relative path format, file existence, external URL/base64 ban) is **deterministically checked** by `scan-design-directory.mjs`. Sub-Agent should write correct paths from the start, but **must NOT spend tool calls verifying image paths** — report any uncertain paths in `qualityWarnings[]` and move on.

Every `<img>` uses `src="../assets/{filename}"` — external URLs, base64 inline, empty `src`, and missing `../` prefix are forbidden. `critical-hero` assets must be used when available; `degraded` assets must be represented with approved CSS degradation, not broken `<img>`. Decorative visual atmosphere should use typography, surface, same-hue texture, or spacing instead of generating filler images.

Deep read trigger: When the page contains complex image/illustration systems, read `aesthetics/index.md §5 Imagery`.

---

## Gate 6 — Density & Crowding

Apply the page's `pageType`:

| Page Type | Required Check |
| --- | --- |
| `showcase` | Visual richness is allowed, but avoid template hero, neon gradients, and unrelated images |
| `information-dense` | First screen must not horizontally cram title, status, stats, and actions into the same readable width |
| `task-driven` | Controls must stay wide and operable; summaries are secondary and narrower |

Deep read trigger: If `qualityRisks` mentions density, wrapping, CJK long title, table, chart, or `libraryRestraintMode: true`, read `operation-policies/page-density-strategy.md`.

---

## Gate 6.5 — Interaction States & Motion

- Primary `<button>`, `<a>`, form controls, clickable cards, tabs, CTA blocks, and media thumbnails cover default/hover/active/focus-visible states where applicable; disabled controls are visually distinct and non-interactive; forms retain clear focus and error states.
- Hover feedback is CSS-first, limited to transform, opacity, clip-path, filter, color/surface/border token steps, or icon movement; `transition: all` is forbidden. Touch devices must not rely on hover for essential content — use always-visible labels or tap-compatible states.
- `@media (prefers-reduced-motion: reduce)` must downgrade non-essential animations to near-zero duration.
- Every page includes at least one business-relevant micro-interaction (ordinary pages: CTA/icon/card feedback; showcase pages: scroll reveal, count-up, subtle parallax).
- Third-party animation libraries are forbidden for ordinary hover/button/card feedback; use them only under the Animation Library Exception in `html-implementation.md`.

Deep read trigger: If the page feels static, if `motionIntensity >= 3`, or if any external animation library is considered, read `aesthetics/index.md §1.5 Motion` and `html-implementation.md` "Animation Library Exception".

## Gate 7 — Anti-AI-Slop

[AVOID] high-frequency low-quality patterns (style-level, non-blocking — but they fail this aesthetic gate):

- Blue-purple neon gradients, rainbow gradients, holographic or light-trail effects
- Colored rounded icon backgrounds for every card
- Decorative icons or eyebrow/kicker/overline text above headings — see `visual-zero-tolerance.md` ❹ (SSOT)
- Generic "modern office / happy people / abstract background" images unrelated to page business semantics
- Same visual section structure repeated across every page without business reason
- Three equal feature cards as the dominant first content block immediately after the hero, unless the business explicitly requires equal comparison
- Hero or first-screen pill/tag overload: more than 5 pills/tags without real information value
- Cards-inside-cards-inside-cards: visible-container definition and depth limits per `sub-agent-hard-rules.md §Nesting-Limit`

Deep read trigger: For brand/marketing Showcase home pages, read the full `aesthetics/self-check.md` before completion.

---

## Gate 7.5 — Hard Visual Defects (Zero Tolerance)

Any single violation below is an **immediate gate failure**. These are the most common visual defects and must be checked before completion.

### Language Consistency

Enforce `visual-zero-tolerance.md` ❶ (SSOT): one language for all user-facing text; only brand proper nouns, universal abbreviations (KPI, VIP, ID, App), and untranslatable technical terms are exempt.

### Component Spacing & Overlap

- [FORBIDDEN] Adjacent cards, buttons, tags, or text blocks with zero visual gap — minimum `gap-2` (8px) required.
- Absolutely-positioned badges/overlays must declare explicit offsets (`top`/`right` ≥ 8px), and the sibling text container must reserve matching padding so text cannot run under the positioned element.
- [FORBIDDEN] Two absolutely-positioned elements sharing a parent without explicit non-overlapping offsets.

### Heading / Eyebrow / CTA Text

Enforce all items in `visual-zero-tolerance.md` (SSOT for heading/eyebrow/CTA text rules): heading length and sentence ban (❸), nothing above headings (❹), button/CTA single-line + length limits (❺).

## Gate 7.6 — Query Generation P0 Guardrails

These checks target high-frequency Query-only generation defects and apply to all pages, especially free-explore / no-Library flows.

### Alignment & CTA Copy

- Buttons, tags, form controls, card titles, list rows, table/toolbars, and nav items in the same visual group must share a clear left edge, right edge, center line, or baseline. Mixed accidental alignment inside one group fails.
- Adjacent controls must use explicit `gap-*`, grid gap, or flex gap. Zero-gap button/tag clusters fail even when they do not overlap.
- Full-width rows must not contain only one small orphan element floating in empty space because of arbitrary padding/margin. This is especially strict for `project-shell` and `shared-branch` parent fragments. A single chip, tab, icon button, small CTA, or tiny KPI may occupy its own row only when it is deliberately aligned to the shared grid/column or intentionally centered as a focal control. Otherwise the row must be completed with related content (title, metadata, status, action, tab group, filters, or summary value) or collapsed into the adjacent row.
- Heading and CTA brevity follows `visual-zero-tolerance.md` ❸/❺ (SSOT). Long user requirement phrases must be split into heading/body plus a short action label.
- [FORBIDDEN] Putting a full sentence into a button, pill, tab, or primary CTA.
- Completion JSON must include `alignmentEvidence` naming the alignment mode for every relevant region (header, toolbar, form row, card/list/table group, nav/tab group) and `headingCtaEvidence` confirming headings are short noun phrases and CTA/button/tab/pill labels are concise. Missing or generic evidence fails this gate.

### Primary Color Discipline

- Each project/page must have exactly one primary brand hue for main CTA and primary selected states.
- In free-explore mode, CSS variables named `--color-secondary`, `--color-accent`, `--secondary`, `--accent`, or their light/dark scales are forbidden. Generate only one brand primary scale.
- If status colors are needed, they must be named `--state-success`, `--state-warning`, `--state-error`, and `--state-info`, and may only be used for real status, validation, destructive actions, or status badges.
- Family roles, identities, categories, content types, avatars, cards, progress ownership, recommendation/comment/share types, and book genres must not use separate green/amber/purple/blue color systems. Use neutral tints, icons, labels, layout, or the single primary tint instead.
- More than one saturated CTA color family on the same page fails unless the user's brief explicitly requests a multi-brand palette.
- Large adjacent high-saturation color blocks fail unless supported by the selected Design Library or reference material.

### Typography Discipline

- Chinese pages must use a sans-serif default stack unless the user explicitly requests editorial, publishing, culture, or serif styling. Accidental Chinese serif/sans-serif mixing fails.
- Even when serif styling is justified, use exactly one CJK serif family for display/title roles and one body stack. Mixing multiple serif faces across hero headings, section headings, cards, and nav makes titles feel "dropped in" and fails. Decorative Latin serif names must not sit before the CJK serif in Chinese heading stacks.
- Use at most 3 primary font weights per page. Randomly mixing many weights or making title hierarchy depend only on boldness fails.
- CJK headings must have readable line-height (`1.18-1.3` for large headings); oversized tight-line titles fail.

### Radius & Cards

- Ordinary pages use the 2 / 4 / 8 / 16px radius scale. Cards, buttons, inputs, and panels must not exceed 16px unless the user explicitly asks for a soft/playful style or the Design Library requires it.
- Full pill radius is limited to avatars, tags, segmented controls, and true pill buttons.
- Habitual `rounded-2xl`, `rounded-3xl`, or large rounded cards across the page fail in ordinary SaaS/tool/task pages.
- Card-in-card decoration (per `sub-agent-hard-rules.md §Nesting-Limit`) and unnecessary containers fail even when spacing is technically valid.

### Shadow & Depth

- Ordinary/static card, table, list, section, and panel shadows must be very subtle: every shadow color alpha must be `<= 0.05`. Prefer borders or surface layering for static surfaces.
- Deep shadows or shadow alpha above `0.05` are reserved for real floating layers only: modals, drawers, popovers, dropdowns, tooltips, toasts, and menus.
- Colored shadows, heavy black shadows on static cards, and border + shadow stacking on ordinary cards fail.

### Chart Generation

- For chart/data pages, prefer Chart.js via the existing `--charts` head injection or a Design Library chart component when available.
- [FORBIDDEN] Hand-drawing complex SVG charts with misaligned labels, arbitrary axes, or text that overlaps chart geometry.

Deep read trigger: If any item above is hard to judge, read `aesthetics/index.md` §1.2, §1.3, §1.4, and §2.5 before completion.

## Gate 8 — Dark Mode Consistency

> **[Skip condition]**: When `<html class="light">` or no class, skip this gate.

When `<html class="dark">`: [FORBIDDEN] pure #000000 as `--bg`, pure #FFFFFF as text color, or light-mode shadow values (they disappear in dark backgrounds). Cards use border or elevated surface instead of shadow-only. Contrast ratio ≥ 4.5:1 against `--surface`, not against `--bg`.

Deep read trigger: If uncertain about dark-mode token values, read `aesthetics/dark-mode.md`.

---

## Gate 9 — Footer & Navigation

Every page in a multi-page project includes a `<footer>` with real content (not placeholder "Link 1 / Link 2"). Navigation active states use exactly one emphasis method (not border + color + weight simultaneously).

Deep read trigger: If footer structure is complex or page has sidebar/mega-menu, read `aesthetics/navigation-footer.md`.

## Gate 10 — Mini Program Chrome

> Applies when `orchestration-summary.json.pages[].miniProgramStyle === true`. SSOT: `html-implementation-mobile.md` "WeChat Mini Program Style Adaptation".

- Page implements the standard mini program nav bar from the SSOT: left back/balancing area, centered title, visible right system capsule (more action + divider + close action with `aria-label` and a close icon such as `data-lucide="x"`).
- [FORBIDDEN] Empty capsule/menu placeholders or invisible right-side spacer as the final mini program chrome.
- [FORBIDDEN] Brand-styled system chrome (`var(--primary)`, `var(--secondary)`, `var(--accent)`, `var(--card)`, or brand shadows).

## Gate 11 — Design Intent Execution

- Every page must have a clear `visualNorthStar` from the task input or current page record, and the final page must visibly express it through hierarchy, spacing, imagery, and section emphasis.
- When `designRead` and `designDials` are provided, the page must reflect them as secondary execution hints without overriding Library, reference, token, responsiveness, accessibility, or validation constraints.
- Showcase / Landing / brand pages must select and execute one `compositionPattern` from `aesthetics/index.md §2.1` or `§11 Showcase Pattern Atlas`.
- [AVOID] Defaulting to `Hero + 3-column Feature + CTA` without a business-specific reason and a named composition pattern.
- Multi-page projects must preserve at least 2 `continuityAnchors` across pages, such as Header/Footer treatment, CTA style, surface layering, type rhythm, radius/shadow discipline.
- Multi-page projects must follow `orchestration-summary.json.project.sharedProjectShellContract` when present. Navigation/header/sidebar/footer structure, primary color system, type stack/scale, radius scale, surface/depth model, CTA style, and alignment rhythm must not drift page-by-page.
- When `orchestration-summary.json.project.generationTree` exists, every leaf page must follow its `generationPath`: inherited fragments are assembled in order, shared regions are preserved unchanged except declared `mutableSlots`, and only declared `privateRegions` are newly generated. If two sibling pages share an ancestor fragment, their shared header/sidebar/body shell must be structurally identical after slot substitution.
- Multi-page/state work must have a materialized `{designProjectPath}/generation-tree.json` before dispatch. The file must use nested `root.children[]` structure and contain all shared nodes and every page/state leaf. A flat top-level `nodes[]` array or root-only tree is incomplete and fails this gate.
- Generation-tree dispatch must be sequential by dependency. A leaf or child branch may start only after every inherited `partials/*.html` file exists. If a page report has `qualityGate: "blocked"` with `blockedReason` containing `missing inherited fragment`, the Main Agent must generate the missing parent node first and then re-dispatch the child.
- Only the Main Agent may dispatch generation-tree nodes. Sub-Agents must not launch child Sub-Agents. A child node may start only after the parent Sub-Agent completion has been received, the parent output file exists and is non-empty, and `generation-tree.json` has been updated with the parent status as `generated`.
- A parent/shared node with `status: "generated"` is not enough to complete a branch. Before consolidation, validation, or final summary, every descendant leaf under that parent must also have completion evidence and a non-empty output file. If any child/descendant is incomplete, the Main Agent must wait/poll and retry; after retries, the parent/subtree is blocked and the incomplete child IDs must be reported.
- Leaf pages must not duplicate parent-owned regions. If a parent fragment owns a breadcrumb, summary header, tab bar, sidebar, top header, footer, or root frame, the leaf page may include that fragment once and replace declared slots only. Repeating the same order ID/customer/header/tab markup in the leaf body is a gate failure.
- Active nav/tab state in shared chrome must be implemented through parent slots, not by per-page header rewrites. If a shared header/sidebar/tab bar has page-specific active states, the parent must provide stable `data-nav-key` or `data-tab-key` attributes, one `[data-active="true"]` style, and an `activeNavItem` / `activeTab` slot. A leaf that duplicates, reorders, restyles, or rewrites the shared nav/tab DOM just to set its active tab fails Gate 11.
- Completion JSON must include non-empty `designIntentEvidence` with `visualNorthStarApplied`, `compositionPatternUsed`, `continuityAnchorsApplied`, and `antiSlopCheck`.
- Completion JSON must also include non-empty `interactionStates`, non-empty `alignmentEvidence`, non-empty `headingCtaEvidence`, non-empty `motionEvidence`, explicit `sourceContextPreserved`, explicit `generationNodeId`, explicit `htmlWriteMode`, explicit `headManagementEvidence`, explicit `inheritedFragmentsUsed`, explicit `privateRegionsGenerated`, explicit `sharedFragmentsPreserved`, and explicit `animationLibrariesUsed` (use `[]` when none). If any third-party animation library is used, `animationLibrariesUsed` must include the library name and a concrete reason that matches the Animation Library Exception.
- For comparison pages derived from an existing page, `sourceContextPreserved` must name at least 3 preserved source-page elements and explain how the requested new state/change is embedded in or layered over that context. `"not applicable"` is allowed only for pages not derived from an existing source page.
- For pages with `stateGroupId`, completion JSON must include `stateGroupId` and non-empty `sharedShellPreserved`. `sharedShellPreserved` must name at least 5 preserved shell regions from the state group's `sharedShellContract`, such as frame, header/sidebar/nav, summary area, control bar, page background, spacing scale, typography scale, radius/shadow model, and component language.
- Derived state pages (`stateRole: "derived"`) must prove they copied the base HTML before editing. Completion JSON must include `derivedFromHtmlSrc` and mention the copied base file in `sharedShellPreserved` or `sourceContextPreserved`. If a derived page ran `fill-html-head.mjs`, created a fresh skeleton, rebuilt the shell independently, or lacks evidence of copying `derivedFromHtmlSrc`, Gate 11 fails even when the final page visually resembles the base.
- If `derivedFromHtmlSrc` is not available on first attempt, the derived task must poll/retry before copying. A missing base file after retries must be reported as `qualityGate: "blocked"` with `blockedReason: "missing base state html after retry: <path>"`; it must not be repaired by generating a fallback skeleton.
- Pages in the same `stateGroupId` may differ only in active control styling and declared `mutableRegions`. If a sibling state changes shared copy, summary cards, navigation placement, spacing rhythm, typography hierarchy, or shell styling without explicit user instruction, Gate 11 fails.
- State siblings must have identical outer shell after allowed slot substitution: `<body>` layout wrapper, outer `<main>` style/class attributes, content wrapper/max-width, header/sidebar/nav, summary/identity region, control/tab bar geometry, tab/control-to-panel seam, panel wrapper padding/background/border, page background, spacing scale, typography scale, and radius/shadow model. A state page with `<main style="padding: 16px">` while its base state uses `<main style="padding: 0">` is a Gate 11 failure unless `main padding` is explicitly listed in `mutableRegions`.
- Tab, segmented-control, filter-state, and same-screen switcher pages must present the switcher and the state panel as one shared content frame. The switcher must not be an isolated floating strip detached from the panel. Only active/inactive control styling and the declared panel body may differ.
- Overlay/floating-layer state pages must preserve the source/base page context underneath the layer and must register hidden return interactions (`hideEdge: true`) for the backdrop/scrim when clickable plus every explicit close/cancel/back control. If the layer visibly implies dismissal but no matching `data-dom-id` is reported in `domIds`, Gate 11 fails.

Deep read trigger: If the page is a brand/marketing Showcase home page, or if you cannot name the pattern being used, read `aesthetics/self-check.md` before completion.

## Completion Requirement

Return `qualityGate: "passed"` only when all gates above pass and `designIntentEvidence`, `interactionStates`, `motionEvidence`, `alignmentEvidence`, `headingCtaEvidence`, `sourceContextPreserved`, `sharedShellPreserved`, and `animationLibrariesUsed` are complete. Use `"not applicable"` for `sharedShellPreserved` only when the page has no `stateGroupId`. If any gate fails, fix the HTML before reporting completion. Validation-script prohibition: see `sub-agent-hard-rules.md §No-Validation-Scripts`.

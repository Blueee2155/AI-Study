# HTML Technical Constraints (Common)

> Three major sections, all mandatory reading for page generation: ① Technical Constraints, ② Layout Structure CSS Implementation Spec, ③ Wrapping and Overflow Defense. The page skeleton is auto-generated by `script/fill-html-head.mjs`. This document defines the **common technical specifications** for all device types.

**Device-specific supplement** (read the one matching device type): desktop/tablet → `operation-policies/html-implementation-web.md`; mobile → `operation-policies/html-implementation-mobile.md`. For aesthetic and visual quality requirements, see `aesthetics/index.md`.

---

## [FORBIDDEN] Canvas Metadata Leaking into Page UI

Node IDs (`page-pricing`), `canvasData` (x, y, group), `pageIndex`, `version`, and other `.design` fields are internal canvas-engine configuration — **absolutely forbidden** in visible HTML areas. Typical leak forms: node ID in page title (`page-pricing · pageIndex 3` → use business-semantic titles); group number in subtitle/tag (`group 0`, `Page 3` → no technical indices); file name in breadcrumb (`pages/index.html` → human-readable nav text); residual debug info (`id="page-pricing"` as visible text → DOM attribute only).

**Self-check standard**: no text that would confuse end users — if it can only be understood by developers, do not render it.

## Tech Stack

- **Tailwind CSS v4** (Browser CDN `@tailwindcss/browser@4`) — used for layout/spacing/responsiveness. Tailwind **semantic** color classes (`bg-primary`, `text-foreground`, `bg-card`, etc.) are allowed via the `@theme inline` bridge; Tailwind **palette** colors (`bg-blue-500`, etc.) are forbidden
- **Icons** — Library-bound: Design Library SVG via CSS mask-image (token-colorable, `currentColor` default); Lucide as fallback. Free-explore: Lucide only (`lucide@1.7.0` CDN).
- **brand CSS file** (`colors_and_type.css`) — brand-prefixed CSS variables (colors/fonts/border-radius/shadows/spacing/typography classes), inlined via `<style id="theme-vars">` (`fill-html-head.mjs` auto-reads and embeds; defines `:root` light and `.dark` variable sets). **[FORBIDDEN]** importing via `<link rel="stylesheet">` — the canvas SDK renders via iframe srcdoc and cannot load external relative paths

## Theme Configuration

- Colors use brand-prefixed CSS variables (e.g., `var(--volcano-primary)`), **not via Tailwind color classes**
- Fonts use Library typography class names (e.g., `volcano-h1`, `volcano-body`) or CSS variables (e.g., `var(--volcano-font-primary)`)
- `<html class="light">` or `class="dark"` is controlled by the canvas SDK and **must be statically hardcoded**
- `lang` attribute matches the target audience (`en`, `zh-CN`, `ja`, etc.)

### [FORBIDDEN] Generating Light/Dark Mode Auto-Switch Scripts

**Absolutely forbidden** to write any JS that auto-detects/switches light/dark mode (localStorage reads, prefers-color-scheme detection, dynamically toggling the dark class, etc.). The canvas SDK controls theme rendering via the static `<html class="light/dark">` attribute; dynamic switch scripts conflict with the SDK and override the design-time theme when exported HTML opens in a browser. The `class` attribute must be **statically written**.

### [FORBIDDEN] Breaking `<head>` Style Infrastructure

**Core principle**: When Sub-Agents edit page content, they are **only allowed to operate within the `<main>` tag** — never touch `<head>` or attributes/scripts on `<body>`.

Full protected-element table and Safe Editing Mode (skeleton/derived-state flows, `SkeletonMainOnly` / `FullHtmlReplaceHead` modes, mode-mixing ban): `sub-agent-hard-rules.md §Style-Integrity` (SSOT).

**Post-edit self-check**: After completing `<main>` content editing, verify style infrastructure integrity using the quick self-check method in `aesthetics/self-check.md > Style Integrity` section.

## Responsive Layout (Mandatory)

**All generated HTML pages must be responsive**, regardless of the user-specified target device. "Make a mobile page" / "make a desktop page" means **prioritizing that device's experience**, not designing exclusively for it — the page must remain viewable and usable on phone, tablet, and desktop. When no device is specified, desktop and mobile are equally important.

**Dashboard Mode Exemption**: When `dashboardMode: true`, pages may use fixed-width layouts (`max-w-[1920px]` or `w-screen`); the Responsive Layout requirement does not apply. See `page-density-strategy.md` "Dashboard Mode".

**Breakpoint strategy** — Tailwind mobile-first: default (no prefix) < 640px phone; `sm:` ≥ 640px large phone/small tablet; `md:` ≥ 768px tablet; `lg:` ≥ 1024px desktop; `xl:` ≥ 1280px large desktop.

**Core principles**: all multi-column layouts, image-text side-by-side arrangements, font sizes, and spacing adapt with breakpoints — mobile defaults to vertical stacking and compact spacing; desktop upgrades to side-by-side and generous spacing. Images and containers use fluid widths; fixed pixel dimensions are forbidden.

[FORBIDDEN] Writing only desktop layout without any responsive breakpoints; writing only mobile layout (fixed `max-w-sm` or `w-[375px]`) with desktop showing a single narrow centered column; fixed pixel widths (e.g., `w-[1200px]`) causing horizontal overflow on small screens.

## Style Rules

**Prioritize Tailwind utility classes**; only when the same style combination appears 3+ times within the same page, extract into `@layer components` using `@apply` for reuse.

**[FORBIDDEN] `@apply` Referencing Local Custom Component Classes**

Parameters for `@apply` must all be Tailwind utilities (e.g., `rounded-xl`, `p-5`, `sm:p-6`); they must never reference custom class names defined in the current file's `@layer components` (e.g., `.section-shell { @apply panel-card p-5; }` is wrong — expand all utilities directly instead).

**Reason**: The Tailwind browser runtime looks up each `@apply` token as a utility. A custom class is not in the registry, so the compiler throws `Cannot apply unknown utility class` and **aborts compilation of the entire `<style type="text/tailwindcss">` block** — the page loses all styles, degrading to bare HTML. Statically detected by `scan-design-directory.mjs`.

| Scenario | Correct Approach |
|------|---------|
| Element styles | Tailwind utility classes (not inline `style="..."`) |
| Color expression | Brand CSS variables `var(--{prefix}-primary)` or Tailwind semantic classes `bg-primary` / `text-foreground` (palette colors like `bg-blue-500` and hardcoded values forbidden) |
| Spacing expression | Tailwind standard spacing `p-4`/`p-6`/`p-8` (not hardcoded `p-[37px]`) |
| Custom CSS | Written within `@layer base` or `@layer components` |
| Icons | Library-bound: `<span data-icon style="mask-image:url('{path}'); background-color:var(--token)">` first; Lucide `<i data-lucide="...">` fallback. Free-explore: Lucide only. |

### Section Background Color Continuity

> Complete principle in `aesthetics/index.md §4 Color & Surface > Continuity`.

Technical implementation: Adjacent `<section>` background lightness difference should maintain smooth transitions. Light pages transition between `var(--{prefix}-background)` / `var(--{prefix}-muted)` / `var(--{prefix}-card)`; Dark pages (`.dark` class active) transition between corresponding dark-mode variables.

### [AVOID] Background Color Sandwich Structure

> Complete principle and forbidden list in `aesthetics/index.md > [FORBIDDEN] Background Color Sandwich Structure`.

**Technical detection method**: From the first `<section>` inside `<body>` to the last `<section>`, check each section's background color CSS variable sequentially. If a sequence of "colorless (`var(--{prefix}-background)` / no background) → colored (`var(--{prefix}-primary)` or other hue-bearing variables) → colorless" appears, it must be corrected to a unified base color scheme.

### Component Background Color Harmony

> Complete principle in `aesthetics/index.md > Component Background Harmony`.

### [AVOID] Neon Colors and Cross-Hue Gradients

> Complete forbidden list in `aesthetics/index.md > [FORBIDDEN] Forbidden Hues`.

Technical alternatives: Neon gradients → same-brand-color opacity gradients (e.g., `var(--{prefix}-primary)` to semi-transparent) or same-hue lightness steps; large high-saturation backgrounds → `var(--{prefix}-muted)` / `var(--{prefix}-card)`.

## Image Rules

| Scenario | Approach |
|------|------|
| Suitable image already in `assets/` | Reference directly |
| No exact match in `assets/` | Preferably select the semantically closest existing image for reuse (Page Sub-Agent must not call image generation tool on its own) |
| External image URL / placeholder image | [FORBIDDEN] |
| Asset status is `degraded` | Do not render a broken `<img>`; use the approved CSS degradation pattern from `sub-agent-hard-rules.md` |
| Image role is `decorative` | Do not create fake image containers; use typography, icon rhythm, surface, spacing, or same-hue texture instead |

**Path**: Pages are in `pages/`; referencing assets **must use relative paths** `../assets/xx.jpg`. [FORBIDDEN] absolute paths, paths missing the `../` prefix, external URLs, or Base64 inline — exported HTML must open directly in the browser, and only relative paths locate the sibling `assets/` directory.

**Image generation constraints**: Image generation is Main Agent scope (Sub-Agents never generate images); prompt wording rules live in `main-agent-repair-flow.md > Image Generation Prompt Constraints`.

**Semantic matching constraint**: Each image's content must be directly related to the business semantics of its containing section. [FORBIDDEN] Using unrelated images to fill image slots. Complete rules in `aesthetics/index.md > Semantic Matching`.

**Image role constraint**: `critical-hero` images are quality-critical and should be used when provided; `shared-brand` images should be reused consistently across pages; `supporting-content` images are optional evidence for specific sections; `decorative` images are not generated by default.

### [FORBIDDEN] Using DOM Elements as Placeholders Instead of Real Images

> Complete forbidden list and placeholder techniques in `aesthetics/index.md > [FORBIDDEN] Using DOM Placeholders Instead of Real Images`.

**Self-check standard**: every `<img>` points to an actually existing file under `../assets/` (all images were pre-generated); no page area "looks like it should be an image but is actually a color block/gradient/icon"; sections that don't need images should not have image containers.

## JavaScript Restrictions

**JS must not modify DOM tree structure** (the design system relies on static DOM parsing).

Forbidden: `createElement`, `appendChild`, `removeChild`, `innerHTML =`, `remove()`, `document.write`, etc.

Allowed: Toggling CSS classes (`classList.toggle`, but not dark/light mode switching), modifying inline styles (`style.opacity`), IntersectionObserver, event listeners that modify styles.

### Animation Library Exception

CSS-first is the default. Do not import animation libraries for ordinary hover, active, focus, button, card, nav, or form feedback. External animation libraries are allowed only when the page explicitly benefits from richer motion and the effect cannot be expressed cleanly with CSS plus small native JS:

| Library Type | Allowed Use |
|------|------|
| GSAP CDN (`gsap@3`, optional ScrollTrigger) | Complex timeline sequencing, scroll reveal choreography, count-up/data reveal, path/text rhythm animation |
| Lightweight particle/canvas background library | Brand/showcase hero atmosphere when business meaning requires "life" or generative motion |

Constraints: only on brand/showcase/landing pages or `motionIntensity >= 4` (ordinary SaaS/dashboard/task-driven/information-dense pages default to CSS-only motion); JS must still not modify DOM tree structure — libraries may update classes, inline transform/opacity/filter, textContent for count-up, or canvas pixels only; particle/canvas effects sit behind content, never covering text/controls or becoming the primary information carrier; add `@media (prefers-reduced-motion: reduce)` or an equivalent runtime guard; completion report must include `animationLibrariesUsed` with each library name and a concrete reason (`[]` when none).

## Allowed External Resources

| Resource | Status |
|------|------|
| brand CSS file (`colors_and_type.css`) | Must include |
| Tailwind CSS CDN | Allowed (only for layout/spacing/responsiveness) |
| Lucide Icons CDN | Allowed |
| Google Fonts and other font CDNs | Allowed (when Library CSS references non-system fonts) |
| Chart.js CDN (`chart.js@4`) | Conditionally allowed: only when orchestration-summary marks the page as `chartsRequired: true` |

### Chart.js Usage Constraints

When `chartsRequired: true`: Chart.js `<canvas>` does NOT violate "JS must not modify DOM tree structure" (Canvas 2D manipulates the pixel buffer, not DOM nodes); Chart.js is restricted to chart/graph rendering only — [FORBIDDEN] using it for animations or non-chart purposes; maximum 6 charts per page (split into multiple pages if more are needed); each `<canvas>` must declare explicit `width`/`height` attributes or be wrapped in a responsive container with defined dimensions.

# Layout Structure CSS Implementation Spec (Common)

## Hero / Display Heading Container Fit

Hero and Display-scale headings must respect actual container width to prevent ugly multi-line fragmentation. **Mandatory CSS on all Display/H1 headings**: `style="text-wrap: balance; word-break: keep-all; overflow-wrap: break-word;"`.

**Layout decision matrix**:

| Heading Position | Max Font-Size Role | Rationale |
|---|---|---|
| Full-width section (≥ 80% viewport) | Display `clamp(40px, 6vw, 88px)` | Ample width for any reasonable title |
| Split column 50-60% | H1 `clamp(28px, 3.5vw, 48px)` | Prevents >3 lines for CJK ≤ 14 chars or Latin ≤ 28 chars |
| Split column < 50% | H2 `clamp(22px, 2.4vw, 32px)` | Very narrow — large text is physically impossible |
| Card / sidebar | H3 `20px` | Constrained context |

**Structural alternatives when Display scale is needed but column is narrow**: place the heading full-width ABOVE the split columns; use an asymmetric split where the text column gets ≥ 65% width; reduce heading character count.

[FORBIDDEN] Display-scale font-size in a container < 500px computed width at 1280px viewport; any heading rendering > 3 lines at its actual font-size; omitting `text-wrap: balance` on Display/H1 headings.

---

## Title Block Layout Constraints

> Design principle in `aesthetics/index.md > Title-Subtitle Combination Layout Principles` and `> Title Block Horizontal Alignment`.

Title blocks adopt a "flexible info side + fixed auxiliary side" dual-column structure: info side allows truncation but not wrapping; auxiliary side must not be compressed; main/subtitle font-size gap ≥ 2 levels; in three-element layouts, title + action on the first line, subtitle on the second.

[FORBIDDEN] `flex-wrap` for title blocks; titles without overflow protection (`min-w-0`); main/subtitle font-size gap < 2 levels; all three elements horizontally aligned on the same line.

### Information-Action Dual-Column (Bottom Action Bars, List Action Rows, etc.)

> Design principle in `aesthetics/index.md > Generalized Applicability of Information-Action Dual-Column`.

All "left-info right-action" layouts follow the same structure — info side stretches flexibly and truncates overflow text; action side is fixed and must not be compressed; buttons keep restrained sizes to avoid encroaching on info space.

[FORBIDDEN] Bottom bar buttons using large pill shapes causing width inflation; info area without overflow protection allowing long text to push buttons; info area multi-line text unrestricted causing bar height inflation.

**[Badcase] Action side width inflation → info side wrapping → empty space below action side** (action element can be `button` / `a` / `div`; root cause is the action side actively encroaching, not "info side unprotected"):

| Violating structure signature | Verdict | Fix action |
|---|---|---|
| Action element in an info-action row simultaneously has ① `rounded-full` (pill), ② font-size at or near the info side main title level, ③ generous padding | All three combined → excessive horizontal volume → violation, even before any visible wrapping | Downgrade action side: `rounded-full` → `rounded-lg`; font-size reduced to subtitle level; padding tightened to compact level |
| Info side text unexpectedly wraps multi-line while the fixed-height action element shows large blank space below it | Layout already failed | Reduce action side specifications — do NOT expand the info side container |
| Action side font-size/height ≥ info side main title | Preventive judgment failure | Action button font-size and height must be visibly smaller than the info side main title |

# Wrapping and Overflow Defense CSS Implementation Spec (Common)

## Text Overflow Mandatory Handling (All Devices)

In any context with width constraints (grid columns, cards, sidebars, flex rows, narrow viewports), **all text nodes must proactively declare truncation behavior; allowing overflow or unexpected wrapping is forbidden.**

**Text types that must be covered**:

| Text Type | Example Scenarios | Required Class |
|----------|----------|------------|
| Single-line title / card main title / list item name | Article title, product name, username, project name | `truncate` (= `overflow-hidden text-ellipsis whitespace-nowrap`) |
| Single-line auxiliary text | Timestamp, tag, status description | `truncate` |
| Card summary / body paragraph | Content description, introduction, detail summary | `line-clamp-{n}` (typically `line-clamp-2` ~ `line-clamp-4`) |
| Fixed-line-count list item body | Notifications, message list | `line-clamp-2` |
| Dynamic data with uncertain width | User input content, API return fields, username, address, notes | `truncate` or `line-clamp-1`/`line-clamp-2` |
| Table cell content | Name column, description column | `truncate` (single-line) or `line-clamp-2` (allow two lines) |

**Assessment method**: For each text node, determine whether its containing container has a width constraint (`max-w-*`, `flex-1`, `w-*`, `grid` sub-cell, etc.). **As long as a width constraint exists, a truncation class must be explicitly declared.**

**Truncation direction**: Tail truncation (`truncate` / `line-clamp-n`) is the default; when head-and-tail abbreviation is explicitly needed, combine with the `direction` property, but no-truncation approaches must not be used.

**Parent container must cooperate**: In horizontal flex layouts, text children declaring flex expansion (`flex-1`) must have the parent (or child) set `min-w-0`; otherwise the child's intrinsic size expands the parent container and truncation declarations become ineffective.

**Post-generation self-check list**: ① all `<p>`/`<span>`/`<h*>` nodes have truncation classes in width-constrained contexts; ② all text sibling nodes with `flex-1` have `truncate` or `line-clamp-n`; ③ card titles/summaries in grid items and all text in sidebars / narrow columns (`col-span-3` and below) default to truncation classes; ④ table name/description columns have `truncate` or `max-w-*` + `truncate`; ⑤ all dynamic-content text (username, product name, description) defaults to truncation classes

[FORBIDDEN] Text nodes within width-constrained containers not declaring truncation classes; using only `overflow-hidden` without pairing with `text-ellipsis whitespace-nowrap` (text is clipped without ellipsis indicator); relying on "natural container truncation" or "generous space" on desktop as justification for omitting truncation classes; multi-line text using only `overflow-hidden` without `line-clamp-n` (truncation position unpredictable).

## Inline Element Overflow Prevention (Tags / Badges / Tag Groups)

Tags, badges, status markers, filter tag groups, and other small inline elements are high-risk areas for wrapping and adaptive anomalies.

**Individual tag/badge**: Must add `whitespace-nowrap` to ensure no line-break; padding restrained to `px-2 py-0.5` ~ `px-3 py-1`; use fixed height (`h-6` ~ `h-8`) + `inline-flex items-center` for alignment.

**Tag groups/filter bars**: Container uses `flex flex-nowrap overflow-x-auto gap-2` for horizontal scrolling; each tag uses `shrink-0` with content-adaptive width.

**Pill shape downgrade** (principle in `aesthetics/index.md §1 Design Tokens > Pill Shape Space Applicability`): Chinese > 4 characters must not use `rounded-full`; change to `rounded-lg`; buttons within cards, sidebars, and mobile action areas default to `rounded-lg`.

[FORBIDDEN] Tag groups using `flex-wrap` causing multi-row tag walls to overflow; tags with fixed equal-width causing short text to stack vertically; tags without `whitespace-nowrap` causing line-breaks; using large padding to inflate tag volume and encroach on content space.

## Data Metric Cards (KPI / Statistics)

KPI numbers use `whitespace-nowrap` (reduce font-size on mobile); description text uses `truncate` or `line-clamp-2`; cards set `min-w-[120px]` to prevent grid compression. On mobile with multiple cards side-by-side, use horizontal scroll (`flex flex-nowrap overflow-x-auto`) rather than forced stacking.

## Table Cell Wrapping Control

Tables are a disaster zone for Chinese text wrapping. Short-text columns (date/ID/status, ≤ 8 chars) add `whitespace-nowrap`; long-text columns set `min-w-[120px]` + `break-words`; each column gets `min-w-*` by content type (data ≥ 80px, text ≥ 120px, action ≥ 60px); headers add `whitespace-nowrap`; on mobile, multi-column tables use `overflow-x-auto`; when columns ≥ 5, prefer `table-fixed` + explicit width ratios.

[FORBIDDEN] All columns equally distributed with no `min-w` causing vertical stacking; fixed-width table containers with no horizontal scroll causing overflow; cell content tags without `whitespace-nowrap`.

## Table Notes Column and Long Text Columns

Notes/recommendation free-text columns set `min-w-[160px]` + `break-words` + `line-clamp-3`. Column width priority: core data columns > identifier columns > notes columns (elastic buffer absorbs remaining space). Primary and secondary information within cells must have visual hierarchy difference.

[FORBIDDEN] Notes columns without `min-w` and `break-words` causing vertical stacking; unlimited notes lines causing table height to spiral; notes columns equally allocated with data columns; primary/secondary info without hierarchy difference.
